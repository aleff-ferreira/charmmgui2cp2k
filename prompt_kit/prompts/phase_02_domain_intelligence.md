# Phase 2 Prompt: Domain Intelligence And Problem Definition

```text
Phase 2 objective: gather credible domain intelligence, define the target/problem space, and select benchmarks, controls, validation criteria, and success metrics.

Project placeholders:

- [PROJECT_NAME]:
- [DOMAIN]:
- [OBJECTIVE]:
- [WORKSPACE_PATH]:
- [RUN_DIRECTORY]:
- [TOTAL_TIME_BUDGET_REMAINING]:
- [REQUIRED_DELIVERABLES]:
- [RISK_CONSTRAINTS]:
- [PREFERRED_OUTPUT_FORMATS]:

What to do:

1. Read Phase 0 and Phase 1 outputs.
2. Use credible sources appropriate to [DOMAIN]: official docs, peer-reviewed literature, standards, primary project documentation, validated datasets, or authoritative technical references.
3. Cite sources with enough detail for a human to verify them.
4. Define the target/problem space in operational terms.
5. Define success metrics that are measurable and tied to [OBJECTIVE].
6. Select benchmarks and controls. Include negative controls or sanity checks where applicable.
7. Define validation criteria and acceptance thresholds.
8. Identify domain-specific failure modes and how they will be detected.
9. Update the assumption register with claims supported by sources versus hypotheses.

What not to do yet:

- Do not install or commit to tools solely because they are popular.
- Do not generate candidates.
- Do not run expensive validation.
- Do not present source-backed general guidance as proof that this project is correct.

Files to create/update:

- `domain_brief.*`
- `sources.*` or `bibliography.*`
- `benchmark_control_plan.*`
- `success_metrics.*`
- `validation_criteria.*`
- `assumption_register.*`
- Current phase log and command log.

Evidence to collect:

- Source URLs/DOIs/manual sections, version/date information, benchmark rationale, control rationale, and explicit acceptance thresholds.

Gate to pass:

Phase 2 passes only if the target/problem is precise, success metrics are measurable, benchmarks/controls are relevant, and key domain claims are supported or clearly labeled provisional.

If tools fail or evidence is insufficient:

- Use alternate credible sources.
- Mark uncertain claims as provisional.
- Narrow the objective or request human review if no credible validation framework can be defined.

End with:

- Phase 2 gate decision.
- Problem definition.
- Sources used.
- Benchmarks, controls, metrics, and validation criteria.
- Open domain questions.
- Whether Phase 3 is authorized.
```
